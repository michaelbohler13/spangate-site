Frequently Asked
Questions

Common questions about setup, the agent, SSH credentials, and device monitoring.

Device Status
Why are my devices showing as Unknown after importing? ▼

Unknown means the agent hasn't pinged the device yet. After a CSV import the agent needs one polling cycle — usually 2–5 minutes — to pick up the new device list and run its first ping sweep.

If devices are still Unknown after 10 minutes and the agent is showing as online, restart the agent container:

Raspberry Pi / Linux
sudo docker restart spangate-agent

After a restart the agent re-fetches the device list immediately and begins pinging. Statuses typically update within 2–3 minutes.

Tip: A restart is most commonly needed after a full wipe-and-reimport (deleting all devices and adding them back). Normal imports — adding new devices to an existing list — usually resolve on their own within a few minutes.
The agent is online but ping statuses aren't updating — what's wrong? ▼

The most common cause is that the agent is busy processing a large number of SSH backup requests and the ping cycle is running slowly. This can happen after a bulk import where many devices were added with SSH enabled at the same time.

Wait a few minutes for the backup queue to drain, then hit ↻ Refresh on the dashboard. If statuses still don't update, restart the agent:

sudo docker restart spangate-agent
A device shows Down but I can reach it — why? ▼

SpanGate uses ICMP ping to determine device status. A few common reasons a reachable device may show as Down:

Firewall blocking ICMP — the device or an intermediate firewall is blocking ping packets from the agent's Pi. Allow ICMP echo requests from the Pi's IP on the target device.

Agent can't reach the device's subnet — the Pi needs a routed path to the device. Check that the Pi's default route or a static route covers the device's subnet.

Stale status — click ↻ Refresh or wait for the next polling cycle. The agent pings every few minutes so status updates are not immediate.

Agent
How do I restart the agent? ▼

SSH into the machine running the agent and run:

Raspberry Pi / Linux (Docker)
sudo docker restart spangate-agent

To confirm it restarted successfully:

sudo docker ps

The STATUS column should show Up X seconds after the restart.

How do I check the agent logs? ▼

View live agent output with:

sudo docker logs spangate-agent --tail 50 -f

Press Ctrl+C to stop following. Look for lines starting with [PING], [SSH], and [HEARTBEAT] to confirm each subsystem is running.

The agent shows as offline — how do I troubleshoot? ▼

First confirm the container is running:

sudo docker ps | grep spangate-agent

If it's not listed, start it:

sudo docker start spangate-agent

If it keeps stopping, check the logs for errors:

sudo docker logs spangate-agent --tail 100

Common causes: invalid API key in the config file, no internet access from the Pi, or a misconfigured backend URL. Check /config/config.yaml and confirm the API key matches what's shown in Settings → API Key.

SSH Credentials
What's the difference between Site Default, Credential Profiles, and per-device credentials? ▼

SpanGate uses a three-tier credential system. The agent uses the first match it finds:

TierWhere to set itBest for
Per-deviceEdit device → Custom (per-device)One-off devices with unique credentials
Credential ProfileSettings → SSH Credential ProfilesGroups of devices sharing the same login (e.g. all core switches)
Site DefaultSettings → Default SSH CredentialsCatch-all for all other devices — set it once and forget it

If a device has no per-device credentials and no profile assigned, it automatically falls back to the Site Default. You only need to set what's different from the default.

SSH backups aren't triggering — what should I check? ▼

SSH config backups require a paid plan and SSH to be enabled on the device. Check the following:

1. SSH is enabled — Edit the device and confirm the SSH toggle is on.

2. Credentials are set — The device needs valid credentials via Site Default, a Credential Profile, or per-device credentials. Test the connection using the Test Connection button in the device edit modal.

3. The agent can reach the device on port 22 — Confirm SSH is open on the device and reachable from the Pi's network.

4. Request a manual backup — Click the backup icon (📋) next to the device. The agent picks up backup requests within 60 seconds.

Import & Export
Can I assign credential profiles during a CSV import? ▼

Yes. Include a credential_profile_name column in your CSV with the exact name of the profile (case-insensitive). The import matches the name to an existing profile and assigns it automatically.

Leave the column blank (or omit it) to use the Site Default for that device.

hostname,ip,vendor,...,credential_profile_name core-sw-01,10.1.1.1,cisco,...,Core Switches access-sw-01,10.1.2.1,aruba_cx,...,Access Layer firewall-01,10.1.1.254,cisco,...,

Download the current template from ⬆ Import CSV → Download Template to see all available columns.

Why doesn't my exported CSV include SSH passwords? ▼

Passwords are intentionally never included in exports — this prevents credentials from accidentally ending up in shared files, email attachments, or version control.

When re-importing a previously exported file, leave the ssh_password column blank and use Credential Profiles or the Site Default to supply credentials to devices that need SSH access.
